# エアコン設定温度アプリ 要件定義書

## 1. 概要・目的

その日の外気温・湿度に応じた推奨エアコン設定温度と運転モードを提示するアプリ。

最適化の優先順位を次のとおり定義する。

1. 快適性（主目的）：室内の体感を快適域に置く設定温度を算出する
2. 健康（制約）：室内外の温度差の抑制、高湿度時の熱中症予防を制約として快適性の算出値を補正する
3. 省エネ（補足）：主推奨を上書きせず、別案として併記する

## 2. スコープ

### MVP範囲
- 現在地または設定地点の外気温・湿度の取得
- 推奨設定温度と運転モードの算出・表示
- 健康制約による補正
- 省エネ別案の併記
- 寒がり／暑がりの個人オフセット設定
- 「今暑い／寒い」フィードバック入力による簡易オフセット調整

### 対象外（将来フェーズ）
- スマート家電連携による自動制御（Nature Remo、SwitchBot、メーカークラウド等）
- 室温・湿度センサの実測フィードバック（実測を用いた閉ループ制御）
- 家族・複数ユーザー共有

## 3. 前提・制約

- アプリの役割は「推奨設定温度の表示のみ」。実際の操作はユーザーが行う（フィードフォワード制御）。
- 実際の室温・湿度は取得しない前提。室内環境の個体差はフィードバック入力による個人オフセットで補う。
- 技術構成は Next.js + Firebase（既存構成を踏襲）。

## 4. 機能要件

| ID | 機能 | 内容 |
|----|------|------|
| F-01 | 気象データ取得 | 地点の外気温・湿度を取得しキャッシュする |
| F-02 | 推奨設定温度算出 | 外気温・湿度から推奨設定温度を算出する |
| F-03 | 運転モード判定 | 冷房／暖房／除湿／送風／不要を判定する |
| F-04 | 健康補正 | 室内外温度差の上限適用、高湿度時の除湿優先 |
| F-05 | 省エネ別案 | 主推奨に対する省エネ寄りの別案を併記する |
| F-06 | 個人オフセット | 寒がり／暑がりのオフセットを設定・反映する |
| F-07 | フィードバック | 「暑い／寒い」入力でオフセットを微調整する |

## 5. ロジック仕様

### 5.1 不快指数（DI）

体感の基準として不快指数を用いる。

```
DI = 0.81 * T + 0.01 * H * (0.99 * T - 14.3) + 46.3
```

- T：気温（℃）
- H：相対湿度（%）

快適域の目安は DI 70〜72。値は運用でチューニングする（未決事項 D-01）。

### 5.2 冷房時の設定温度算出

目標室内 DI と想定室内湿度から、設定温度 T を逆算する。5.1 の式を T について解くと次のとおり。

```
T = (DI_target - 46.3 + 0.143 * H_in) / (0.81 + 0.0099 * H_in)
```

- DI_target：目標室内不快指数（初期値 71）
- H_in：想定室内湿度（初期値 55%。冷房により外気湿度より低下する想定）

### 5.3 暖房時の設定温度算出

暖房時は湿度の寄与が小さいため、目標室温ベースで算出する。

- 基準室温：20〜22℃
- 個人オフセットを加算して最終値とする

### 5.4 運転モード判定

外気温の閾値でモードを切り分ける。閾値は初期値であり調整可能とする。

| 外気温 | モード |
|--------|--------|
| 25℃以上 | 冷房 |
| 18〜24℃（中間期） | 除湿／送風／不要（湿度で分岐） |
| 18℃未満 | 暖房 |

中間期は湿度が高ければ除湿、低ければ送風または「エアコン不要」を提示する。

### 5.5 健康補正（制約）

快適性の算出値に対し、健康ロジックを制約として適用する。

- 室内外温度差の上限：冷房時、設定温度の下限を `外気温 - 7℃` でクリップする（寒暖差による負荷の抑制）。快適性の算出値がこの範囲内なら補正しない。
- 高湿度時の除湿優先：外気湿度が 70% 以上のとき、除湿運転を優先提示する（熱中症予防）。

快適性を主目的とするため、健康補正は快適域を逸脱する場合のみ作用するガードレールとして機能する。

### 5.6 省エネ別案（補足）

主推奨を上書きせず、別案として併記する。

- 夏：主推奨 +1℃
- 冬：主推奨 -1℃
- 参考値として環境省の目安（夏28℃／冬20℃）を表示可能とする

### 5.7 個人オフセット

最終設定温度に個人オフセット（±1〜2℃）を加算する。フィードバック入力（「暑い」「寒い」）の累積でオフセットを簡易調整する。

## 6. データモデル（Firestore）

### users/{uid}/settings
- location：地点（地点コードまたは緯度経度）
- offset：個人オフセット（℃、初期値 0）
- comfort_di_target：目標室内DI（初期値 71）
- indoor_humidity：想定室内湿度（初期値 55）

### weather_cache/{location}/{timestamp}
- temp：外気温（℃）
- humidity：相対湿度（%）
- fetched_at：取得時刻

### users/{uid}/feedback/{id}
- created_at：日時
- signal：hot / cold
- temp_at_feedback：入力時の外気温

## 7. 外部データ（気象API）

全機能が同一ソースに依存するため、抽象化レイヤ（インタフェース）を挟み、ソースを差し替え可能にする。呼び出し制限対策として取得結果を Firestore にキャッシュし、1日数回更新する。

候補（未決事項 D-02）：

| 候補 | 長所 | 留意点 |
|------|------|--------|
| 気象庁 | 無料・APIキー不要・日本の信頼性が高い | エンドポイントが非公式・地点が区域コード単位 |
| OpenWeatherMap 等 | 緯度経度指定・体感/湿度/風を一括取得 | APIキー必要・日本の地点粒度と無料枠を要確認 |

必要データ：外気温、相対湿度。将来的に体感補正へ拡張する場合は風速も取得する。無料枠・エンドポイント・地点粒度は実装時に確認する。

## 8. 画面構成

- メイン：今日の推奨（設定温度・運転モード・省エネ別案）、外気温・湿度の表示
- 設定：地点、寒がり／暑がりオフセット、目標DI等のパラメータ
- フィードバック：「今暑い」「今寒い」ボタン

## 9. 非機能要件

- オフライン時はキャッシュした直近データで表示する
- パラメータ（DI目標値、閾値、マスタ）は設定またはコード上で調整可能にする
