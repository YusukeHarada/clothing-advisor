# 服装・衣替えアプリ 要件定義書

## 1. 概要・目的

現在の気温・天気に応じて、当日の服装の厚さ指標と衣替え時期を提示するアプリ。

対象は性質の異なる2機能で、両者を同等に扱う。

- 当日の服装（連続・毎日利用）：その日の予報から服装の厚さを提示する
- 衣替え時期（離散・季節イベント）：気温推移からクローゼットの入替え時期を判定する

## 2. スコープ

### MVP範囲
- 地点の気温・天気・風速・湿度の取得
- 当日服装の提示（昼／朝晩の二段表示）
- 直近気温との相対提示（「一枚多め／少なめ」）
- 衣替え判定（週間平均気温の移動平均、ヒステリシス付き）
- 寒がり／暑がりの個人オフセット設定

### 対象外（将来フェーズ）
- 衣替え提案のプッシュ通知
- 手持ち衣類の登録・コーディネート提案
- 家族・複数ユーザー共有

## 3. 前提・制約

- 技術構成は Next.js + Firebase（既存構成を踏襲）。
- 相対提示・衣替え判定・順応が日次気温の時系列履歴に依存するため、履歴保存を初期から設計する。
- 衣替え判定に7日分の予報が必要。データソースは気象庁を採用する（決定ログ D-02）。

## 4. 機能要件

| ID | 機能 | 内容 |
|----|------|------|
| F-01 | 気象データ取得 | 地点の気温（最高／最低）・天気・風速・湿度・7日予報を取得しキャッシュする |
| F-02 | 体感温度補正 | 風速・湿度・日射・雨から補正気温を算出する |
| F-03 | 当日服装提示 | 補正気温を服装レベルに変換し、昼／朝晩の二段で表示する |
| F-04 | 相対提示 | 直近平均との差分から「一枚多め／少なめ」を提示する |
| F-05 | 衣替え判定 | 週間平均気温の移動平均で衣替え時期を判定・提案する |
| F-06 | 個人オフセット | 寒がり／暑がりのオフセットを服装レベルに反映する |
| F-07 | 気温履歴保存 | 日次の気温・湿度・予報を蓄積する |

## 5. ロジック仕様

### 5.1 体感温度補正

気温を風速・湿度・日射・雨で補正し、補正気温を得てから服装レベルに変換する。

- 風速：Missenard 等の体感式で風による低下を反映する（風速が大きいほど体感気温を下げる）
- 湿度：高湿度時は夏の暑さを増幅、冬の寒さを増幅する方向で補正する
- 雨：防水の要否フラグとして扱い、服装アドバイスに付加する。フラグは体感補正の内部値に留めず、8章の画面構成で傘・レインウェアの要否として明示表示する
- 日射：晴天時は日中の体感を上げる方向で補正する

MVPでは風速・湿度による補正を実装し、日射・雨は段階的に追加する。

### 5.2 服装レベルマスタ

補正気温を段階レベルに変換する。マスタは Firestore に持ち、後から調整可能とする。低温・高温の両端は体感差が大きいため区分を細分化し、初期値は次のとおりとする。

| レベル | 気温帯 | 服装例 |
|--------|--------|--------|
| 0 | 35℃以上 | 猛暑対応（半袖・通気重視、日射/熱中症注意） |
| 1 | 25〜34℃ | 半袖 |
| 2 | 22〜24℃ | 長袖1枚 |
| 3 | 18〜21℃ | 重ね着（カーディガン等） |
| 4 | 15〜17℃ | 薄手の上着 |
| 5 | 12〜14℃ | コート |
| 6 | 8〜11℃ | 厚手コート |
| 7 | 0〜7℃ | 防寒（マフラー・手袋等） |
| 8 | 0℃未満 | 厳寒対応（防寒強化・氷点下想定） |

### 5.3 当日服装（表示単位）

日較差に対応するため、原則は昼と朝晩を分けた二段表示とする。

- 昼：最高気温を補正した値のレベル
- 朝晩：最低気温を補正した値のレベル

`users/{uid}/settings.outing_hours`（外出時間帯）が設定されている場合は、二段表示に加えて以下を適用する。

- 外出時間帯が昼夜どちらか一方に収まる場合（例：日中在宅・朝夕のみ外出）：該当する時間帯の予報気温を補正したレベルを主表示とし、二段表示は補助情報として縮小表示する
- 外出時間帯が未設定の場合：従来どおり昼／朝晩の二段表示のみを主表示とする

外出時間帯に対応する時間別予報の取得可否はデータソース（D-02）に依存するため、取得できない場合は最高／最低気温からの近似（例：朝晩枠は最低気温、昼枠は最高気温）で代替する。

### 5.4 相対提示（順応の反映）

同じ気温でも直近の気候により体感が変わる（順応）。直近平均を基準線とし、当日との差分でアドバイスを付す。

- 基準線：直近3日間の平均気温
- 当日の補正気温が基準線より一定以上高い／低い場合に、5.2のレベル表と連動した具体的な差分を提示する（例：「レベル3相当だが、直近より暖かいためレベル2相当の服装で可」）。「一枚多め／少なめ」のような抽象的な文言のみでなく、レベルラベルに基づく服装例を併記する

絶対気温ベースの服装レベル（5.3）に、相対差分の補正層（5.4）を重ねる二層構成とする。

### 5.5 衣替え判定

単日でなく週間平均気温の移動平均で判定し、頻繁な切替提案を防ぐためヒステリシスを設ける。

- 判定値：直近7日の平均気温の移動平均
- 秋→冬方向：移動平均が下限閾値を継続して下回ったら「厚手への入替え」を提案
- 春→夏方向：移動平均が上限閾値を継続して上回ったら「薄手への入替え」を提案
- ヒステリシス：上げ局面と下げ局面で閾値を変え、境界付近での往復提案を防ぐ
- 日付の区切りはJST（日本標準時）を基準とする
- 閾値・ヒステリシス幅を運用変更した場合、既存の`changeover_state`（現在の衣替え状態）はリセットせず、変更後の閾値で継続判定する

閾値の初期値は次の目安で仮決めし、運用で調整する。

- 秋→冬方向：移動平均が15℃を継続して下回ったら提案
- 春→夏方向：移動平均が20℃を継続して上回ったら提案
- ヒステリシス幅：2℃（上げ局面・下げ局面で閾値を2℃ずらす）

衣替え提案は対象外機能（プッシュ通知）を持たないMVPでは見逃されやすいため、該当時はバナー表示に加えて「今日の服装」画面の最上部に固定表示し、ユーザーがアプリを開けば必ず気づける導線とする。

### 5.6 個人オフセット

寒がり／暑がりのオフセットを服装レベルに反映する。オフセットは「弱／中／強」の3段階で設定可能とし、それぞれ弱0.5段／中1.0段／強1.5段でレベルを調整する。初期値は「中」（レベル1.0段）とする。

## 6. データモデル（Firestore）

地点は`locationId`で正規化し、将来の複数拠点・家族共有（対象外機能だが移行コストを下げるため今から設計に反映）に備える。

### locations/{locationId}（マスタ）
- name：表示名
- geo / area_code：データソースに応じた地点識別子（緯度経度 or 気象庁区域コード等）

### users/{uid}/settings
- locationId：主たる地点（`locations/{locationId}`を参照。旅行等で一時的に変更する場合は`temporary_location_id` + `temporary_until`を別途保持し、期限切れで自動的に主地点へ戻す）
- offset：個人オフセット（強さ：弱／中／強、初期値 中）
- outing_hours：外出時間帯（任意、5.3の表示切替に使用）

### clothing_levels/{level}（マスタ）
- temp_min / temp_max：気温帯
- label：服装例

### weather_history/{locationId}/history/{date}
- temp_max / temp_min：最高／最低気温
- humidity / wind / weather：湿度・風速・天気
- forecast_7d：7日予報（衣替え判定用）
- fetched_at：取得日時（キャッシュ鮮度表示に使用）

### users/{uid}/locations/{locationId}/changeover_state
- current_season_mode：現在の衣替え状態（例：夏物／冬物）
- last_suggested_at：直近の提案日時

地点単位で状態を持つことで、複数拠点対応時にユーザー設定側の変更のみで拡張できる。

## 7. 外部データ（気象API）

全機能が同一ソースに依存するため、抽象化レイヤを以下の3層で挟み、ソースを差し替え可能にする。

1. **LocationResolver**：ユーザーの地点情報（住所・緯度経度等）を、データソースが要求する識別子（気象庁の区域コード、OWMの緯度経度等）に変換する
2. **WeatherProvider（取得層）**：ソース固有のAPI呼び出しを担い、ソースごとのレスポンス形式を吸収する
3. **正規化DTO**：取得結果を「最高／最低気温・天気・風速・湿度・7日予報」の共通スキーマに変換し、以降のロジック層（5章）はこのDTOのみに依存する

衣替え判定と相対提示が7日予報と履歴を要求するため、ソースは7日以上の予報を無料枠で提供する必要がある。

取得はクライアント起動時ではなくサーバ側（Cloud Functions等）のスケジュール実行に統一し、1日数回、単一経路でFirestoreに書き込む。クライアントはFirestoreの値を読むのみとし、同時アクセスによる書き込み競合を避ける。

採用：**気象庁**。無料・APIキー不要・週間予報ありを決め手とする。地点はLocationResolverで区域コードに変換して扱う。

検討した候補：

| 候補 | 長所 | 留意点 | 採否 |
|------|------|--------|------|
| 気象庁 | 無料・APIキー不要・日本の信頼性が高い・週間予報あり | エンドポイントが非公式・地点が区域コード単位 | 採用 |
| OpenWeatherMap 等 | 緯度経度指定・7〜8日予報・体感/風/湿度を一括取得 | APIキー必要・日本の地点粒度と無料枠を要確認 | 不採用 |

必要データ：最高／最低気温、天気、風速、湿度、7日予報。エンドポイント・地点粒度（区域コード⇔ユーザー地点の対応表）は実装時に確認する。

当日内の予報更新（例：午後からの降雨・気温急変）については、スケジュール取得の更新回数（1日数回）の範囲で反映されるが、更新間隔・通知手段はMVP対象外とする。ユーザーには表示中データの取得時刻（`fetched_at`）を明示し、情報の鮮度をユーザー自身が判断できるようにする。

## 8. 画面構成

- 今日の服装：昼／朝晩（または外出時間帯基準）のレベルと服装例、相対アドバイス、傘・レインウェアの要否表示、データ取得時刻
- 衣替え：判定状態と提案（該当時はバナー表示に加え画面最上部に固定表示）
- 設定：地点（一時的な地点変更を含む）、寒がり／暑がりオフセット（強さ選択）、外出時間帯

## 9. 非機能要件

- オフライン時はFirestoreの直近キャッシュデータで表示し、取得時刻を明示して鮮度を示す（具体的なオフライン保持方式はNext.jsのService Worker/IndexedDB活用を実装時に確定）
- 服装レベルマスタと衣替え閾値はコードまたは設定で調整可能にする
- 気温履歴は、衣替え判定（直近7日＋ヒステリシス確認用バッファ）と順応判定（直近3日＋バッファ）に必要な日数を満たすため、最低30日分を保持する

## 11. 補足：エアコンアプリとの共通基盤

体感温度・不快指数の計算（風速・湿度補正を含む）、地点特定（LocationResolver）、寒がり／暑がりオフセット、気象データ取得・キャッシュ（WeatherProvider・正規化DTO）は、エアコン設定温度アプリと共通化できる。将来的に単一アプリへ統合する場合は、これらを`packages/weather-core`のような共有モジュールとして切り出す。
